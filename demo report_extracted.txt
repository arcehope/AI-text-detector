ABSTRACT 
The Web Application Vulnerability Scanner is a comprehensive security solution which performs automated detection of standard security flaws found in contemporary web software. Websites face growing cybersecurity threats because they store sensitive user credentials and financial data and personal information which attackers seek to exploit. SQL Injection and Cross Site Scripting (XSS) and Open Redirect vulnerabilities and security header problems together create security weaknesses which enable attackers to gain unauthorized system access and cause data breaches and financial and reputation losses. The project intends to identify system vulnerabilities through its execution of automated scanning procedures. The system operates by crawling a target website to gather all accessible pages together with their associated forms and input elements. The system conducts testing by sending specific test payloads through these input fields while patterns and response variations and status codes work to identify any existing vulnerabilities. The system detects SQL Injection through the process of sending specially designed database inputs which reveal error messages or unusual system responses. The system detects XSS vulnerabilities through testing which checks whether dangerous scripts are visible in the system responses. The scanner verifies security settings through checks of HTTP response headers which identify essential security protection elements. The scanner uses an OWASP-based risk assessment system which calculates the potential impact and ease of exploitation and verification strength of each identified security flaw to assign risk ratings that range from Low to Medium to High. The system uses React technology to create its interactive dashboard while the backend scanning engine operates through Python and Flask.
Keywords: Web Security, Vulnerability Scanner, SQL Injection, Cross Site Scripting (XSS), Open Redirect, OWASP Risk Assessment, Ethical Hacking

CHAPTER 1
INTRODUCTION
1.1. Introduction 
In the modern digital age, web applications have become a very important part of various industries, such as the banking industry, the e-commerce industry, the healthcare industry, the education industry, the government industry, etc. These web applications store very sensitive information, such as personal details, login credentials, transactions, etc. As the use of web applications is increasing, the probability of cyber-attacks on the security vulnerabilities of these web applications is also increasing. If there is a slight security vulnerability in a web application, it can result in severe consequences, such as data breach, unauthorized access, financial loss, etc.
Common security issues that can occur in web applications include SQL Injection, Cross Site Scripting, Open Redirect, and security headers. The causes of such security issues in web applications include poor input validation, poor coding practices, and lack of security testing. Performing security testing on a web application can be a complex process, and human error can occur. Thus, automated security solutions are essential.
To overcome this challenge, this project has come up with the creation of the Web Application Vulnerability Scanner, which is considered to be a full-stack security tool that is meant to scan web applications and identify the common web vulnerabilities present in the target website. This is achieved through the crawling of web pages and the identification of the forms and input fields present in the website, and finally, the analysis of the responses to identify abnormal behavior. This is achieved through the application of various techniques, such as pattern matching and the application of risk score mechanisms in conformity with the OWASP guidelines.
The application is built using React as the front-end tool, Python as the back-end tool, and MongoDB as the database tool to store the results and the reports generated from the application. This tool is considered to be helpful in enhancing the security of web applications through the early detection of vulnerabilities, which is considered to be helpful in the development of more reliable web applications.

1.2. Problem Statement
The Web Application Vulnerability Scanner tries to solve the following problems:
This includes web applications in industries such as banking, commerce, health care, and education.
Common web application vulnerabilities include SQL Injection, Cross Site Scripting (XSS), Open Redirect, and security headers.
These security issues can cause problems such as unauthorized access, data breaches, financial losses, and reputational damage.
Manually conducting security tests is a slow, error-prone process that requires specific skill sets, hence inefficient.
There is a need to develop a system that can automatically scan web applications, identify vulnerabilities, and provide guidance on the same.

The main objectives of this project are as follows:

To implement the techniques to detect SQL Injection, Cross Site Scripting (XSS), Open Redirects, and Security Header misconfigurations.
To implement a severity-based risk assessment system based on the OWASP standard.
To design a user-friendly interface to scan the website and view the vulnerability report.
To store the scan results in a database.

CHAPTER 2
BACKGROUND STUDY AND LITERATURE REVIEW 
2.1. Background Study
With the development of online services, web applications have become the backbone of industries such as finance, commerce, health, education, and governments. From online transactions to health records and online learning, web applications are used to handle a large amount of sensitive information. As the use of web applications is increasing, so is the risk of cyber-attacks. Hackers are always looking to find weaknesses in web application systems to exploit them and gain unauthorized access to the system.
Over the years, certain common web vulnerabilities have been identified that are major security threats to web applications. These include attacks such as SQL Injection, in which hackers gain access to confidential data using their malicious queries. Cross Site Scripting allows hackers to run malicious scripts on a user’s machine, stealing their session information or redirecting their requests to malicious sites. Similarly, vulnerabilities in Open Redirects allow hackers to redirect users to malicious sites. Also, missing or incorrect security headers weaken the defensive shield of a web application. These have been identified in web security standards such as the OWASP guidelines in the well-known OWASP Top 10 list.
Normally, web application security testing is done manually by experts using penetration testing. Even though this method is effective in testing web applications, it is also time-consuming and expensive. Additionally, the expertise level of the tester plays a crucial role in the process. Since web technologies are advancing rapidly, using this traditional method to test web applications is not feasible anymore. This has given birth to automated tools that perform vulnerability scanning on web applications. These tools include OWASP ZAP and Burp Suite. However, these tools may also be complex to understand for beginners or expensive for small-scale projects.
The idea behind a Web Application Vulnerability Scanner is based on the idea of using web crawling to navigate through web pages, injecting payloads to test input boxes, using pattern matching to identify malicious activity, and analyzing server responses to compare results. By using these tools, web application vulnerabilities can be easily identified in a matter of seconds and in a structured format.
However, with the evolution of full stack web development technologies like React for the frontend interfaces and Python-based frameworks like Flask and FastAPI, it is possible to develop custom security tools with interactive interfaces and real-time reporting capabilities. Thus, there is a possibility of creating user-friendly and lightweight vulnerability scanning tools with the help of these emerging technologies.
Thus, the increased reliance on web applications and the sophistication of threats emphasize the need for automated security testing tools. A Web Application Vulnerability Scanner is a tool developed based on the principles of security research and automated testing with the aim of offering a structured approach for addressing web vulnerabilities.

As the use of the Internet grows, the number and relevance of web applications have also grown, being an integral part of many sectors and businesses. However this growth has adverse effects in the form of increased security threats. Given the large number of current vulnerabilities and the wide variety of testing techniques and tools used to find vulnerabilities, it becomes complex for software developers and application testers to select the proper tools and techniques to test potential threats. This paper aims to collect and classify current security-oriented software testing tools, techniques, and security development models for web systems. According to the STRIDE threat model, our, classification considers software development activities, and associated security threats. To accomplish our goal, we conducted a systematic literature review (SLR), from 2017 to 2022. We identified 18 software testing techniques, 88 tools and four secure development processes, methodologies or models. We found a great variety of tools and techniques, from traditional penetration testing to state-of-the-art Artificial Intelligence supported tools, and we associate different threats found with their respectively testing techniques and STRIDE classification. We believe that our work serves as a foundation for software testers to select proper and modern techniques, tools, and security models, processes or methodologies related to security testing, in accordance with their threat analysis, potentially improving their security testing for web systems. 3.1.1.1 Functional Requirements

Figure 3.1: Use Case Diagram for Web Application Vulnerability Scanner
1. Login/Registration:
A new user can register for an account or login into the Web Vulnerability Scanner System using their existing account. This will help in keeping user information secure and also give them access to scanning results among other services.
2.  Manage Profile:
Users have the ability to control and modify their personal information like usernames, emails, and passwords. It will make sure that users have correct and updated account information.
3.  Add Target URL:
The URL of the website that the user wishes to scan is provided by the user. Target website details are saved to prepare for scanning the website.
4.  Start Vulnerability Scan:
Vulnerability scans can be initiated for the selected target website by the user. The crawling procedure starts with analyzing web pages in the website to detect vulnerabilities.
5.  Crawl Website:
The system will start crawling the targeted web page through navigation from one linked page to another and gathering data necessary to analyze potential vulnerabilities.
6.  Detect Vulnerabilities:
This process includes scanning the target site for web security vulnerabilities, which include SQL injection, XSS, CSRF, poor authentication, directory traversal, open ports, and security header misconfiguration, among others. This increases the level of safety and risk assessment of the website.
7.  Generate Scan Report:
Once the scanning process is complete, a report detailing the vulnerabilities, their severities, the affected pages, and the solutions to them is created.

8.  View Scan Progress:
The application gives an account of the scanning process, such as the status of scanning, number of pages scanned, problems detected, and estimated time for completion.
9.  View Scan Reports:
The user can look into the previous scan reports to determine the detected vulnerabilities. This is useful in analyzing the website security and keeping record of the vulnerabilities.
10.  View Vulnerability Details:
The user can have details on the detected vulnerabilities by viewing their descriptions, severity, impact, locations affected, and how they can be mitigated.
11.  Manage Users:
Admins can control the use of registered users through various means such as accessing user details, managing user permissions, suspending users' accounts, or even deleting users when required. This will ensure system security and effective user management.
12.  View All Scan Histories:
Admins can track all vulnerability scans conducted within the system. Admins will be able to track scanning processes, user activities, and security monitoring processes.
13.  Monitor System Activities:
Admins can keep track of all system activities that include scans, system logs, user activities, and server status.

3.1.2. Feasibility Analysis
3.1.2.1.  Technical Feasibility
Technical feasibility is proven through the use of modern web technologies and advanced techniques of vulnerability detection to find typical vulnerabilities on websites and web applications. The development is done using FlaskApi for backend processes and MongoDB database technology to store data like user data, scan history, vulnerabilities, and target websites' data. This will make it possible for the system to perform different scans and vulnerability detections efficiently. This is achieved through the use of automated crawling and vulnerability detection to detect common vulnerabilities on websites such as SQL injections, XSS, CSRF, poor authentication, missing security headers, directory traversal and open ports. Lightweight Ness of the scanning tool allows it to function well on standard computers and even cloud servers. The frontend will be developed using React or modern web technologies and APIs will be used to enable communication between the frontend and the backend. The application is designed to be modular, and new vulnerabilities can easily be added.

3.3. Algorithm Details
3.3.1. Web Application Vulnerability Scanning Algorithm (Core System)
The Web Application Vulnerability Scanning Algorithm refers to a hybrid method of assessing the security status of web applications. This methodology works by initiating automated HTTP/HTTPS requests and analyzing responses from the server via various scanning modules. A vulnerability report will be created from these findings.
This system uses a modular approach during which various security assessments are carried out either sequentially or concurrently. These security assessments cover aspects like HTTP header, SSL/TLS configuration, network port, application technology, input validation, access control, and data exposure.
The outcome of the process produces vulnerabilities that are classified based on severity levels and a risk score of the target application. The severity categories include Critical, High, Medium, Low, and Informational.
3.3.2. Rule-Based Detection Algorithm
Rule-Based Detection Algorithm is the one used to detect vulnerabilities based on the use of security rules pre-defined for HTTP responses and applications. The checks done include:
Absence of security headers
Unsecure attributes of cookies
Misconfiguration of CORS policy
Exposure of server details
Access of sensitive files
The rules are considered independently and when any violations of these rules are detected, then that violation is categorized as a vulnerability based on the corresponding severity level. This algorithm helps achieve consistent vulnerability detection.
3.3.3. Payload Injection Algorithm
The Payload Injection Algorithm is what we use to see what happens when a real attack is made on a system. We do this by putting code into the parts of a program where users can put information. The system checks a things:
SQL Injection using bad SQL code like' OR '1'='1
Cross-Site Scripting, also known as XSS using JavaScript code like <script>alert('xss)</script>
After we put in the bad code the system looks at what the server says to figure out:
If there are any SQL error messages
If any bad scripts are run
If the server is acting weird
The Payload Injection Algorithm helps us find problems that happen when a program does not check what users put in well. This is important because the Payload Injection Algorithm can help us make sure the Payload Injection Algorithm is working by using the Payload Injection Algorithm to test it.
3.3.4. Signature-Based Detection Algorithm
The Signature-Based Detection Algorithm detects vulnerabilities using a comparison between server responses and the known signatures of attacks. These include:
SQL error codes (MySQL, Oracle, PostgreSQL error codes)
Signatures of JavaScript execution
Signatures of sensitive files
Signatures of debugging/system errors
When the signature is detected in the response body, it is flagged as a vulnerability.
This approach works well for detecting problems that have been well documented before.
3.3.5. Response Analysis Algorithm
The Response Analysis Algorithm evaluates differences between normal responses and responses generated after payload injection. It analyzes:
Reflected input values 
Unexpected output changes 
Error variations 
Data leakage patterns 
This helps detect vulnerabilities that may not match exact signatures but still behave abnormally under manipulated inputs.
3.3.6. Port Scanning Algorithm (Socket-Based)
Port Scanning Algorithm is used to check whether any ports on the targeted server are open through sockets-based connections. This is done for commonly utilized ports like:
HTTP (port 80), HTTPS (port 443)
SSH (port 22), FTP (port 21)
Database ports (ports 3306, 5432)
Remote ports (port 3389)
These ports are checked by making TCP connections. The port is considered to be opened if the connection succeeds.
Having open risky ports increases the attack surface area of the application.
3.3.7. Severity Scoring Algorithm
The Severity Scoring Algorithm computes the total risk rating of the targeted software by assessing the identified vulnerabilities. These vulnerabilities have their individual severity rating:
Critical
High
Medium
Low
Informational
A weightage for severity calculation is used to derive a total risk rating.
Risk Rating = Critical x 40 + High x 20 + Medium x 10 + Low x 5
Where:
Critical: Number of critical vulnerabilities
High: Number of high vulnerabilities
Medium: Number of medium vulnerabilities
Low: Number of low vulnerabilities
3.3.8. Technology Fingerprinting Algorithm
Algorithm for technology detection finds out technologies used on the target web application based on the analysis of HTTP headers and response body. It finds out the following:
Web server (Apache, Nginx)
Framework (React, Angular, Laravel, Django)
CMS (WordPress, Drupal, Joomla)
It uses signature-based detection for:
Headers in server responses
X-Powered-By headers
HTML source code
It also could find out about version exposure which implies outdated software used.
3.3.9. Directory Brute Force / Enumeration Algorithm
The Directory Enumeration algorithm tries to find out any sensitive file or directory by checking for paths defined in a pre-specified list. They can be, for example:
/admin
/.env
/.git
/backup
/phpmyadmin
A request is made for each path, and the status code obtained from it is analyzed:
200 → The accessible resource
403 → Access forbidden
301/302 → The resource got redirected
This helps identify exposed sensitive files or misconfigured access control.
3.3.10. Cookie Security Validation Algorithm
The Cookie Security Validation Algorithm tests the security properties of cookies configured by the application. These include:
The Secure property (cookie should be HTTPS-only)
The HttpOnly property (cookie can’t be accessed via JavaScript)
SameSite property
When one of these properties is not set, then the cookie is deemed to be insecure.
This prevents session hijacking attacks and cross-site attacks.

This model is responsible for validating the provided target URL, performing vulnerability scanning asynchronously, streaming scan progress to the front-end, and saving scan results in the database. Once scan request has been received by the system, the first step in the process is to validate and normalize the URL to make sure that it is correct and that it includes a valid host name along with the proper HTTP or HTTPS scheme. As each scan progresses, individual scan modules report progress updates to a message queue, which streams the scan progress back to the user through the SSE channel. 4.1.2.3. HTTP Security Header Analysis Model
The HTTP Security Header Analysis Model is intended for assessing the existence of necessary security HTTP headers in server responses of the target web application. In the beginning of the scanning procedure, the scanner sends an HTTP request to the target URL via the configured session object and receives the set of response headers from the target server. Then the module compares the obtained headers with the preconfigured list of necessary security headers, including, but not limited to, Strict-Transport-Security, Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and Permissions-Policy. In case if any necessary header is not present, the scanner creates a structured vulnerability report that includes, among other components, the name of the vulnerability, its severity, description, detailed information, and recommended solution. The model provides a possibility to monitor the status of the scanning procedure in real time due to using callback methods that inform the system when the scan is started and completed. In case if any networking or requests-related errors are occurred during the analysis, the system catches these exceptions and creates an information finding instead of stopping the scan.

The SSL/TLS Certificate Validation Model measures the level of security associated with the HTTPS connection by forming a secure connection between the socket and the target host server and acquiring its SSL certificate. CORS Misconfiguration Detection Model assesses the behavior of the targeted web application when it comes to managing Cross-Origin Resource Sharing policy to see whether cross-origin requests are restricted in a secure way. The mechanism performs an HTTP request to the target web application using a specially crafted origin header and then analyzes the received response to examine Access-Control-Allow-Origin and Access-Control-Allow-Credentials response headers. The presence of the wildcard origin indicates that this configuration can be considered a medium or high security risk if credentials allow. CHAPTER 5
CONCLUSION AND FUTURE MANAGEMENTS 
5.1. Conclusion
In summary, the Web Application Vulnerability Scanning System project demonstrates an effective implementation of automated security analysis for modern web applications. The system is designed to identify common web vulnerabilities such as missing security headers, SSL/TLS weaknesses, open ports, SQL Injection, Cross-Site Scripting (XSS), insecure cookies, CORS misconfigurations, directory exposure, and information disclosure issues. By integrating multiple scanning modules into a single platform, the system provides comprehensive security assessments and helps users detect potential risks before they can be exploited by attackers.
The project follows a modular and scalable architecture using Flask for backend development, MongoDB for data storage, and React with Vite for frontend implementation. Authentication and authorization are securely managed using JWT-based authentication, while bcrypt hashing ensures password security. The system also supports real-time scan progress streaming using Server-Sent Events (SSE), enhancing user experience by providing live vulnerability scanning updates.
The implemented scanning modules successfully meet the functional requirements of identifying vulnerabilities, generating risk scores, maintaining scan histories, and producing structured scan reports. The system demonstrates technical feasibility through the use of lightweight and efficient technologies, while operational feasibility is achieved through an intuitive user interface and automated scanning workflow.
As cybersecurity threats continue to evolve, this project provides a strong foundation for improving web application security awareness and vulnerability management. Overall, the Web Application Vulnerability Scanning System contributes significantly to secure software practices by enabling developers and organizations to proactively identify and mitigate security weaknesses in their applications.

5.2. Future Recommendation
The Web Application Vulnerability Scanning System has strong potential for future improvements and scalability. The following enhancements are recommended for future development:
Integrate Advanced Vulnerability Detection: Add advanced security testing features such as CSRF detection, command injection testing, authentication bypass analysis, and API security scanning.
AI-Based Risk Analysis: Implement machine learning techniques to predict vulnerability severity and prioritize risks based on attack patterns and historical scan data.
Real-Time Threat Monitoring: Extend the system to support continuous monitoring and real-time alert notifications for newly discovered vulnerabilities.
PDF and Dashboard Reporting: Add professional PDF report generation and interactive analytics dashboards for easier vulnerability analysis and reporting.
Multi-Threaded and Distributed Scanning: Improve performance by implementing distributed scanning architecture for handling large-scale web applications efficiently.
User Role and Team Management: Introduce role-based access control and collaborative team features for enterprise-level vulnerability management.
Cloud Deployment and Containerization: Deploy the system using Docker and cloud platforms to improve scalability, portability, and production readiness.
Integration with Security Tools: Integrate the scanner with tools such as Nmap, OWASP ZAP, or Burp Suite for deeper and more advanced security assessments.
